## 起因

我在 Cursor 和 Claude Code 上装了很多 Agent Skill。

但说实话——**基本没用过**。不知道能干什么，偶尔想到一个需求，技能也不会自动出现。

我以为是我用法不对。于是做了一次彻底的排查。结果比我想象的更荒唐。

---

## 第一个发现：我装了 156 个技能

三个目录加起来：

| 目录 | 数量 |
|------|------|
| ~/.cursor/skills/ | 15 |
| ~/.agents/skills/ | 67 |
| ~/.claude/skills/ | 74 |
| **总计** | **156** |

156 个。我自己都愣了一下。

哪来的？有些是主动装的，有些看到推荐顺手装的，有些是技能市场一键安装的。装的时候想着"说不定用得上"——然后就再也没打开过。

---

## 第二个发现：30+ 个重复安装

很多技能在两三个目录里都装了一份。比如 `find-skills` 同时存在于 Cursor、Agent、Claude 三个目录。`markdown-to-html` 也是。`skill-creator` 也是。

光去重就能砍掉 28 个。

---

## 第三个发现：大量"概念性"技能

什么叫概念性技能？举个例子。

**brainstorming-0.1.0** 的全部内容大概是：

> 帮用户把想法转化为完整的设计和规格说明。先理解项目上下文，然后逐个问题引导用户细化想法……

没有调用任何 API，没有运行任何脚本，没有操作任何工具。**它只是一段教 AI 怎么说话的提示词。**

而 AI 本身就会头脑风暴。装不装这个，效果没有区别。

**copywriting-0.1.0** 也一样——内容是 AIDA 公式、PAS 框架、FAB 方法论。这些信息 AI 的训练数据里早就有了。安装这个，等于给一个已经会开车的人发了一本《如何开车》的说明书。

这类技能大约有 70 个。共同特点：**全是文字，没有执行能力。**

---

## 第四个发现：Skill 的触发机制是"关键词匹配"

这是最关键的一个认知。

当你在 Cursor 里发一条消息，系统会扫描所有已安装 Skill 的 `description` 字段，和你说的话做匹配。匹配上了，那个 Skill 的 SKILL.md 就会被注入到 AI 的上下文中。

所以，**一个 Skill 能不能被触发，完全取决于它的 description 写得好不好**。

我那些触发率高的技能，description 都是这么写的：

```
当用户说「质检工单」「检查工单」「归纳工单」「分类工单」
或直接提供工单号（如 T202603050182）时触发。
```

```
当用户说「下班了」「收工」「今日总结」时自动触发。
```

直接列出中文触发词，简单粗暴但有效。

而那些永远不被触发的技能，description 长这样：

```
>
```

或者：

```
|
```

是的，就一个符号。连描述都没有，系统拿什么去匹配？

---

## 所以问题出在哪？

装了 156 个技能却用不起来，不是 AI 不行，不是技能系统有 bug，而是**三个结构性问题叠加**：

### 1. 数量太多，匹配精度下降

156 个技能同时竞争匹配，AI 需要在一堆候选中找到最合适的。候选越多，误匹配和漏匹配的概率越高。就像你家有 156 把钥匙挂在墙上，每次开门都要试半天。

### 2. 大部分技能没有真正的执行能力

概念性技能就是一段提示词，AI 本身就具备这些能力。它们占着位置，增加噪音，但不产生价值。

### 3. 不知道技能的"触发暗号"

每个技能都有自己的触发关键词，但用户不知道。你说"帮我分析一下这个数据"，但数据分析技能的 description 写的是 "SQL, pandas, and statistical analysis expertise"——中英文都对不上，自然触发不了。

---

## 我做了什么

### 大清洗：156 → 50

用了一个下午，把 156 个技能逐个分类评估：

- **删除重复**：28 个只保留一份
- **删除概念性**：约 70 个纯提示词技能
- **删除未配置**：需要 API Key 但从未配置的
- **删除不相关**：A 股分析、学术论文、Slack GIF 之类与我工作无关的

最终保留 50 个有明确用途的技能。精简 67%。

### 做了一个"元技能"

清洗之后我做了一个叫 `show-my-skills` 的技能——**一个管理所有其他技能的技能**。

说"我的技能"就会列出所有技能，按使用频率排序，展示每个技能能做什么和怎么触发。它还会自动追踪每个技能的使用次数，用久了就形成一份个性化的技能使用画像。

再也不会出现"装了不知道有"的问题。

## 一些感悟

### 少即是多

50 个精选技能，比 156 个杂乱技能有用得多。候选池小了，匹配精度高了，AI 找对技能的概率大幅提升。这不是什么新道理，但在工具这件事上，我们总是要亲自踩一遍。

### 工具的价值不在于拥有，而在于使用

这和买课、囤文章、收藏教程是同一个道理。装 156 个技能，给人一种"我很强"的错觉——但从来不用，它就只是硬盘上的几个文件夹。

拥有感是假的。使用才是真的。

### 好的技能需要好的"接口设计"

一个技能好不好用，50% 取决于核心功能，50% 取决于 description 写得好不好——也就是触发词设计。这和写 API 文档一样：功能再强，接口定义不清，别人就是用不起来。

### Agent Skill 系统还很早期

当前的"关键词匹配"触发机制相当原始。理想状态应该是：AI 理解意图后，自动选择最合适的技能组合完成任务，而不是靠用户记住每个技能的暗号。

但在那一天到来之前，能做的只有：**精简数量、明确触发词、知道自己有什么。**

---

## 实操建议

如果你也装了一堆 Agent Skill 但觉得没用，可以试试：

1. **盘点**：列出所有已安装的技能，分类评估
2. **去重**：多个目录重复的只留一份（Cursor > Claude > Agent 优先级）
3. **删概念**：纯提示词、没有执行脚本的技能，删掉不影响任何功能
4. **查配置**：需要 API Key 的技能，不用就删，用就配好
5. **学暗号**：记住常用技能的触发关键词，或者做一个像 `show-my-skills` 这样的索引
6. **控总量**：保持在 30-50 个以内，宁缺毋滥

记住：**技能不是越多越好，是越准越好。**

工具是用来用的，不是用来收藏的。
